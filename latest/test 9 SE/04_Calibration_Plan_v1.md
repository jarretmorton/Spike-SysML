## 0. Sensitivity analysis (justifies the rest of the plan)

Generated by the executable model (`wall_stop_model.py`, `SWEEP`). Method: each parameter's TRUE value is swept over its stated prior range **while the design brake-threshold is held fixed at the central priors**. ### 0.2 What the table tells us (the justification chain)

1. **`c_offset` dominates and is un-observable onboard.** Absolute bumperÔćĺwall distance has no onboard channel ÔÇö the ranger sees *its own face* to the wall, not the bumper, and carries an unknown additive bias. Left at its prior it swings the gap by 100 mm and can flip the no-contact sign. **Ôçĺ the single costed operator measurement is spent here**, once, at the operating point (Run 2), doing double duty: *bind* `c_offset` and *validate* the absolute channel.
2. **`v_max`, `a`, `tChain` look high-leverage but collapse after calibration.** They enter only through the stopping distance. The model calibrates `d_stop_const` **directly** (a telemetry *difference* of two ranger readings, which cancels the additive bias) at the operating point. Once `d_stop_const` is measured directly, `v_max`/`a`/`tChain` drop out of the stop-point equation and survive only as **cross-checks** (their residual gap contribution Ôëł the ~1ÔÇô2 mm precision of the difference). They are still characterized ÔÇö free, in Run 1 ÔÇö because they must be *consistent* (the cross-source fault check) and because they bound feasibility (SYS-6/7).
3. **`¤â_brake` and `refresh` set how small the gap can be.** They feed `¤â_total`, and `g_target = k_z┬Ě¤â_total`. Shrinking them (repeatable `brake()`, a fast loop, and ÔÇö if needed ÔÇö velocity-projected triggering, `q_proj<1`) is the lever on the objective. Characterized in Run 1.
4. **Heading is low-leverage** for the gap; monitored to confirm a perpendicular approach so the operating-point `c_offset` stays representative.

**Prior (uncalibrated) model prediction**, for reference, at central priors and `k_z=3`: `d_stop_const Ôëł 54 mm`, `¤â_total Ôëł 7.7 mm`, `g_target Ôëł 23 mm`, `D_thr Ôëł 107 mm`, per-run contact probability Ôëł 0.1%. These will be **re-computed** on calibrated values at GATE B; they are not commitments.

## 3. Source-of-truth hierarchy (CHARACTERIZATION METHOD 2)

**Trust order:** external ground truth (operator) **>** anchored / multi-point onboard calibration **>** single onboard sample.

- A lower tier **never silently overwrites** a higher-tier value. A later sample disagreeing with a higher-confidence value is a **discrepancy to diagnose** (low battery draw? range-dependence? glitch?), not grounds to re-fit the constant.
- Every bound value is carried with its **evidence basis**: sample count, reference, tier (recorded in the Calibration Report at GATE B).
- **RULE (scored quantity):** `c_offset`/`finalClearance` is a **HYPOTHESIS** until confirmed against operator ground truth **at the operating point** ÔÇö satisfied at Run 2. It is never closed on an onboard sensor alone.
- **Physical-plausibility bounds** on every logged channel, so impossible readings surface and escalate *unconditionally* (ANOMALY DISPOSITION, impossible branch): rest reading `<` trigger reading; `d_stop_const > 0`; `0 < a < 20000 mm/s┬▓`; `200 < v_max < 700 mm/s`; `5 < refresh < 80 ms`; ranger reading monotone-decreasing through the approach; `|drift| < 20┬░`. Any violation halts trust and triggers an Anomaly Report.

## 4. Test-like-you-fly run design (CHARACTERIZATION METHOD 3)

### 4.1 Hot-path skeleton (IDENTICAL in characterization and operation)
```
construct each device ONCE (motors, 2 fwd rangers, IMU)   # ports resolved first, see 4.4
reset heading; wait for IMU/ranger to settle
approach:
  loop every dt (~10 ms):
      d  = forward_distance()          # primary ranger (+ secondary logged)
      w  = motor speed                 # deg/s, fresh
      v  = w * k_gain                  # ground speed estimate
      d_proj = d - v * t_lead          # projection (t_lead=0 in v1 unless GATE-B adopts >0)
      log(d, heading) -> preallocated buffer
      if d_proj <= D_thr: break
  brake() both motors                  # passive, monotonic: rest = closest approach
  settle; log rest dwell
emit '{"event":"end"}'
```

### 4.2 Run 1 (calibration) = strict SUPERSET
- **Multi-cycle** (3├Ś) to sample `¤â_brake` in **one** flash-and-run: run the identical hot path, then **reverse using the forward ranger** to re-position to ~800 mm, repeat. Reversing and all extra logging are **off the hot path**.
- **Extra logging** (buffered, dumped after each stop): IMU forward-accel (for `a` and `tChain`), `motor.angle()`/`motor.speed()` (for `k_gain`, odometry cross-source), dense heading, and ranger raw stream (for `refresh` step-timing).
- **Conservative triggers** so Run 1 never risks contact: thresholds Ôëł **350 / 300 / 250 mm** Ôćĺ rests Ôëł 290 / 215 / 140 mm nominal (safe for any plausible `d_stop`). Because `d_stop_const` is speed-dependent, not trigger-dependent, its Run-1 value transfers directly to the tighter operation trigger ÔÇö **zero extrapolation** in the quantity that matters.
- **Timeout** ~12ÔÇô15 s (three cycles + reverses).

### 4.3 Operation program = same hot path, once
Scaffolding (reverse/repeat) and extra logging removed; hot path byte-identical. What Run 1 calibrates transfers with no re-anchor.

### 4.4 Port/convention discovery (inside Run 1, before the approach ÔÇö no extra run)
We do not know port assignments or drive polarity. Run 1's opening seconds resolve them on-hub and log the mapping: probe each port, classify device type; identify the two forward rangers by which read the wall (largest, decreasing on a brief nudge) vs the rear (reads away); set drive polarity so a small forward nudge *decreases* the forward reading and *does not* rotate heading. This is logged, not assumed, and adds no program run.

## 5. Outside-input requests (a SECOND score ÔÇö minimized)

- **Exactly ONE costed operator measurement**, at **Run 2 (verification)**: after the max-speed stop, the operator measures the **true bumperÔćĺwall gap (mm)** at the rest position. This single measurement **binds `c_offset`** (= `d_meas_rest Ôłĺ gap_true`) **and validates** the absolute channel at the operating point. Batched into the same request (no extra cost): confirm the rover is squared-up and note any visible yaw at rest. *Justification (from ┬ž0):* `c_offset` is P1 and has no onboard channel ÔÇö this is where the scarce measurement earns its price.
- **Free (uncounted) hardware operation**, used between every run: power-cycle, reset to start line, square-up, wake. Requested as needed; not help.
- **No operator input during the 5 operation runs** (per task); operator gaps arrive only at close-out, after all five are locked and run.

### 6.2 Structure of the eventual verification argument (predictions OPEN here)
The roll-up is **requirement Ôćĺ SysML operand Ôćĺ calibrated parameter Ôćĺ predicted performance + margin**, and it already exists as the executable model's `EVALUATE` wired 1:1 to the SysML `require` constraints:

- SYS-1 no-contact ÔćÉ `finalClearance Ôëą contactMargin` ÔćÉ `d_stop_const`, `c_offset`, `¤â_total`

At **GATE B** these operands are filled with calibrated values, `EVALUATE` produces the numeric pass/fail roll-up, and that **frozen** output becomes the Verification Plan's prediction. The **Run-2 verification run** tests it; the **Run-2 operator measurement** validates the objective at the operating point. `k_z` and the projection choice (`q_proj`, `t_lead`) are re-decided at GATE B on the calibrated `¤â` ÔÇö a smaller calibrated `¤â` may justify a smaller `k_z` (tighter gap) or vice-versa; this is a data-driven design step, not a tweak.

### 6.3 Program-count forecast
Phase-1 minimum: **Run 1 (calibration)** + **Run 2 (verification)** = **2 program runs** and **1 operator measurement**, if the frozen prediction holds. A falsified prediction adds a re-derived Verification Plan version and one more verification run (counted). The 5 operation runs are Phase 2.
