# Test 5 (SE Arm) — One-Page Summary

**Task.** Drive the rover straight at a wall (~1 m away) at maximum speed and stop as close as possible without touching it. Two hard constraints: *run at maximum speed* and *no contact*. The arm followed a gated systems-engineering process — analyze, model, calibrate, verify, then five scored operation runs — touching hardware only after the analysis was done.

**Result.** All five operation runs stopped with no contact, 62–77 mm from the wall (mean 71, best 62), at full speed. The core win: the predict-then-verify step caught a faulty sensor before the scored runs — the verification run exposed it, the trigger was reassigned B→A, and the stop re-derived around the accurate sensor — which is what made the close, safe stops possible. On that reassigned sensor the arm froze, at Gate C, a predicted true rest gap of 78 mm; the five scored runs averaged a true gap of 71 mm (operator-measured ground truth at close-out), so on a like-for-like true-gap basis the prediction held to −7 mm, on the safe side. Getting there took six characterization/verification runs and one human intervention, more than planned, because several real hardware faults surfaced and were fixed rather than worked around. No fault reached the scored runs — though "clean" rested partly on risks that were flagged and not closed: the shipped program was changed twice late and never formally re-verified, and it leaned on a single ultrasonic sensor that can drop out.

**What went wrong, and how it was corrected:**

- **First run spun in place.** The motors are mounted mirror-image, so a same-direction command rotates the rover rather than driving it forward. An early run detected forward direction from the distance reading, which a spin corrupts, so it spun ~3 turns in place — no contact. Fixed: forward direction is now decided from the heading sensor, which tells rotating from traveling.

- **Veer at full speed — active heading-hold.** The two motors don't run at exactly equal speed, so the rover veered. The arm added a closed-loop steering correction that trims the *leading* wheel down (never past maximum, so full speed holds), tuned from the rover's own veer over a few characterization runs. Once tuned it held heading within a few degrees every scored run — by design, not luck.

- **Telemetry stalled the control loop.** An early run sent data out during the fast approach; those writes blocked the loop and starved the sample rate, and on one run the loop briefly froze. Fix: store all readings in memory and send them only after the rover stops.

- **A faulty ultrasonic sensor — found by the human measurement.** The one operator measurement of the true gap (264 mm) exposed a fault. The sensor the program treated as primary read ~100 mm *shorter* than the true gap — impossible, since a sensor cannot read less than the distance to the front of the rover — while the other read slightly *long*, consistent with sitting just behind the front edge. So the "primary" sensor was the faulty one. The arm excluded it, switched the trigger to the accurate sensor, and re-derived the stop around it. That reassignment is what let the rover stop close and safely.

- **Two late changes, never re-verified.** The sequence: a verification run carried a frozen prediction that the then-primary sensor would read 135 mm at rest (band 102–168); it read 166, so the prediction passed. But the operator measurement at that run (264 mm true gap) exposed the faulty sensor, and the program was changed to trigger on the accurate sensor. Re-deriving around that sensor — still before any operation run — the arm set the committed operation prediction: a true gap of ~78 mm (the accurate sensor's calibration plus a 3σ no-contact margin). The changed program was *not* re-verified against that prediction before proceeding (a process error; the arm offered a confirming pass, which the operator declined). The first operation run then stopped cleanly but exposed a timing problem — the faulty sensor's read could stall the fast loop — so a change was requested (read the accurate sensor first). Because the rules require one locked program for all five scored runs, the arm was given a choice: continue with the original program unchanged, or reset the five-run count and make that first run a characterization run. It reset the count and took the fix. That reset run validated the accurate-sensor *stop*, so what actually went unverified was the sensor-read *order*, not the stop.

**Stopping-algorithm changes.** The stop was always triggered by an ultrasonic reading and the stop command (a motor brake and hold) never changed — only *which* sensor decides the stop and *when* it is read. Three versions: trigger on the smaller of the two readings (which the human measurement exposed as the faulty sensor); trigger on the accurate sensor with the threshold and true-gap correction re-derived around it; then read the accurate sensor *first* and fall back only if it drops out, because near the wall the faulty sensor returns no echo and can stall the loop. A standing risk throughout: the trigger relies on a single ultrasonic sensor, and these intermittently drop readings, so a badly-timed dropout would fall back to the faulty one plus a blind-travel guard — a thin net that was never exercised.

**What held by luck.** Two things were safe by fortune, not knowledge. The safety cushion was sized from a couple of stops, not a measured spread, and the real spread came in comfortably under the assumed figure — conservative, but not known-adequate beforehand. And the accurate sensor's offset was anchored far (~264 mm) but applied close (~71 mm); it shrank slightly up close, so the onboard estimate read ~8 mm short of the truth — safe direction, but not guaranteed from one anchor. Caught and quantified at the end.

**Score against the criteria.**
1. Characterization/verification runs (fewer better): **6** — four calibration runs and two verification runs (the second being the first operation run, which was reset to characterization and served as a de-facto re-verification of the accurate-sensor stop), vs a two-run plan.
2. Human interventions (fewer better): **1** — the true-gap measurement at verification, which is what exposed the faulty sensor.
3. Operation runs with no contact (more better): **5 / 5**.
4. Closeness of stops: **62–77 mm, mean 71, best 62; standard deviation ≈ 5.5 mm**.
5. Predicted vs actual error (operation): **71 mm − 78 mm = −7 mm** — 7 mm closer than the committed ~78 mm prediction (derived at the sensor switch — see the sequence bullet), on the safe side.
