# Test 5 (Freestyle Arm) — One-Page Summary

**Task.** Drive the rover straight at a wall (~1 m away) at maximum speed and stop as close as possible without touching it. Two hard constraints: *run at maximum speed* and *no contact*. The freestyle arm was given the task with no prescribed method — it chose its own approach: a few characterization runs to tune the program on the rover's own sensors, then five scored operation runs.

**Result.** All five operation runs stopped with no contact, at full speed — but 180–257 mm from the wall (mean 214, best 180), far short of the tens-of-mm intended. The reason is one clean thing: the arm's onboard distance reading was ~127 mm *shorter* than the true gap, and it never found that out in time. Its own committed estimates put the stops at ~55 mm; they were really ~185 mm. So the rover believed it had stopped close while it actually parked well back — safe, but by a wide margin it could not see. It got there with six characterization runs (one a locked-program validation) and zero human measurements — and that last choice, made to protect the score, is exactly what hid the error.

**What went wrong, and how it was corrected:**

- **A large gap, from a sensor offset the arm never anchored.** During characterization the two forward ultrasonic sensors disagreed by ~127 mm; the arm noticed, used the *smaller* of the two readings as a safe estimate, and deliberately deferred sorting out whether the gap was real — because resolving it would have cost a ground-truth measurement, and it was steering its score to zero outside inputs. It then targeted a ~30 mm sensor reading, reasoning that measuring the start distance with the same sensor would cancel any offset. That holds for the rover's internal book-keeping but not for where it physically stops: the ~127 mm under-read shifted every stop that much farther from the wall. Because no ground-truth measurement was taken during characterization, the offset stayed invisible — every onboard cross-check was self-consistent in the shifted reading-space — so it surfaced only in the mandatory operator measurements *after* the five scored runs, too late to fix. The arm's own retrospective is blunt: one measurement during setup would have removed ~127 mm from every stop.

- **Yaw — an active heading-hold that worked.** Left alone the rover veered ~14°. The arm added a closed-loop steering correction keyed to the heading sensor, applied so it steers without disturbing the straight-line wheel balance, and it held heading tight — within about ±1–4° on every run. Staying straight was by design, not luck.

- **Telemetry stalled the control loop.** An early run sent data out during the fast approach; those writes blocked the loop (up to ~75 ms per cycle) and fired the brake ~31 mm late. Fix: buffer all readings in memory and send them only after the rover stops, which kept the loop fast (~2 ms) and the brake on time. A later run then found the end-of-run data dump itself too slow (a timeout), so the logging was slimmed.

- **Wheel slip, and two far outliers.** The stop is triggered by wheel-rotation count (the ultrasonic is read only once, at the start, because it lags badly at full speed). But the wheels slip on launch and skid under the hard brake, so the wheel count does not perfectly track true travel — adding run-to-run scatter. On two runs a low start-distance reading combined with slip made the rover brake early and stop even farther out (~256 mm), the two worst results.

**Stopping-algorithm changes (top level).** The trigger was wheel-rotation count from the start (the ultrasonic lags at speed, so it is read once at the start and kept out of the fast loop), and the stop command (a motor brake and hold) never changed. The changes across runs were tuning, not structure: buffering the telemetry so the loop would not stall, then calibrating the braking distance and the target down toward a close stop. The one thing the algorithm never corrected was its own reference: it assumed the start-distance reading cancelled any sensor offset, so the ~127 mm under-read rode straight through into every stop.

**What held by luck.** No contact — the paramount goal — held partly by luck. The un-anchored ~127 mm offset happened to point in the *safe* direction: the sensor read *short*, so the rover stopped *farther* than it thought. Had the sensor read long by the same amount, the identical approach would have driven into the wall. On top of that, the arm padded its target conservatively because it was unsure of the sensor-to-bumper offset and the braking spread. So the clean 5/5 rested on a lucky sign and a conservative cushion, not on knowing where the rover actually was.

**Score against the criteria.**
1. Characterization/verification runs (fewer better): **6** — five characterization runs plus one locked-program validation run (the verification run) — vs a ~3-run plan (and one failed host launch that did not execute).
2. Human interventions (fewer better): **0** — no ground-truth measurements were taken during characterization (the choice that kept the sensor offset invisible).
3. Operation runs with no contact (more better): **5 / 5**.
4. Closeness of stops: **180–257 mm, mean 214, best 180, standard deviation ≈ 39 mm (sample)** (the three clean runs were tight, ~7 mm; the two outliers drove the spread).
5. Predicted vs actual error: **none — no prediction was frozen before the runs** (its onboard estimate, a post-stop reading rather than a committed prediction, ran ~155 mm low: 59 vs 214 mm on the clean runs — about **two-thirds of the true gap**, so the rover "believed" it was ~3× closer than it was — and was found only at close-out).
